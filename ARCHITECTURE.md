# Gakuji Architecture

A map of what lives where and why. This describes **roles**, not current
values — it says what `app_text_styles.dart` is *for*, not which font sizes
are in it right now. Values change constantly and would make this doc wrong
within a week; roles change rarely. When you add a new file or change what a
file is fundamentally *for*, update this doc. Changing a font size, a color,
or a list of sample data does not require an update here.

## The shape of the app, in one paragraph

Gakuji is a Flutter app with three tabs (Home / Dictionary / Library) plus
per-deck study flows (reading flashcards and kanji handwriting). Today,
everything is local: decks and folders are in-memory Dart lists, and
progress/preferences live in `SharedPreferences` on-device. Google
Sign-In + Firestore cloud sync is currently being built on top of this (see
"Where this is headed," below) — that work is not merged into `main.dart`
yet as of this writing.

## Entry point

- **`lib/main.dart`** — app startup. Initializes Firebase, kicks off
  background preloading of the dictionary database and the Japanese
  handwriting model (both non-blocking — the UI doesn't wait on them), then
  builds the `MaterialApp`. Also contains a debug-only iPhone preview frame
  and on-screen size/DPR label (`MyApp.useIphonePreviewFrame` /
  `showScreenSizeDebugLabel`) for testing layout without a real device.
- **`lib/firebase_options.dart`** — generated by `flutterfire configure`.
  Per-platform Firebase config. Don't hand-edit; regenerate via the CLI if
  it ever needs to change.

| `library_page.dart` | Library tab — deck list and folder list, with new-deck/new-folder creation dialogs. |
| `folder_page.dart` | Contents of one folder (the decks inside it). |
| `deck_page.dart` | A single deck's home screen — study button, entry to the term list, deck options menu. |

## `lib/theme/`

- **`app_text_styles.dart`** — the single source of truth for text styling
  (`AppText.pageTitle`, `AppText.cardTitle`, etc.). Built this way
  specifically so font sizes live in one place instead of scattered as
  inline `TextStyle`s across every screen. Call sites reference a role
  name and `.copyWith()` overrides for one-off differences (color, weight)
  — they shouldn't invent new inline `TextStyle`s.

## `lib/models/` — plain data classes

- **`term.dart`** — the core model, `Term`. See "The Term/Deck relationship"
  below — this is the one genuinely tricky concept in the whole codebase.
- **`deck.dart`** — `Deck`, plus the `DeckType` (writing/reading/hybrid) and
  `StudyMode` enums.
- **`folder.dart`** — `Folder`. Organizes deck *IDs*; a folder doesn't own
  or copy decks.
- **`writing_point.dart`** — a single `(x, y, time)` point in a handwriting
  stroke.
- **`writing_prompt.dart`** — one writing-study prompt, derived from a
  `Term` (reading/meaning shown, kanji hidden as the answer).

## `lib/data/` — in-memory seed data (not the real dictionary)

- **`deck_data.dart`** / **`folder_data.dart`** — the actual `decks` and
  `folders` lists the app runs on today. This is what the Firestore
  migration will eventually replace.
- **`pinned_deck_data.dart`** — pinned-deck bookkeeping (`pinnedDeckIds`,
  `isDeckPinned`, `canPinMoreDecks`).
- **`recent_searches.dart`** — in-memory recent-searches list for the
  Dictionary tab.
- **`dictionary_data.dart`** — a small **hardcoded fallback word list** (a
  handful of entries). This is *not* the real dictionary — see the
  dictionary pipeline section below for where the real data comes from.
  This file is the emergency fallback if the real database fails to load.
- **`reading_card_edit_data.dart`** — the `ReadingCardEditData` model
  (which glosses/examples/photo a specific card shows) plus its JSON
  (de)serialization, paired with `services/reading_card_edit_storage.dart`.

## `lib/services/` — logic that isn't just a data class

- **`dictionary_service.dart`** — owns the *real* dictionary. Copies the
  bundled SQLite asset to app-writable storage on first run, opens it
  read-only, and runs all dictionary search queries against it. Falls back
  to the small hardcoded list in `dictionary_data.dart` if the database
  can't load.
- **`deck_storage.dart`** — `SharedPreferences`-backed per-deck progress
  and shuffle-state persistence.
- **`reading_card_edit_storage.dart`** — `SharedPreferences`-backed
  persistence for `ReadingCardEditData`.
- **`writing_recognition_service.dart`** — wraps
  `google_mlkit_digital_ink_recognition` for Japanese handwriting
  recognition; manages downloading the on-device recognition model.
- **`writing_answer_checker.dart`** — normalizes and compares a submitted
  handwriting answer against the correct kanji.
- **`prompt_converter.dart`** — tiny `Term` → `WritingPrompt` mapper.

## The Term/Deck relationship — the one genuinely tricky part

`dictionaryWords` (the real dictionary, served via `DictionaryService`) is
the canonical, global list of words — one `Term` per unique dictionary
entry. These are never meant to be mutated.

When a word gets saved into a deck, `Term.deckCopyFrom(...)` creates an
**independent copy** with a brand-new `id`, while `sourceId` keeps pointing
back to the original dictionary term's id. This is why:

- Marking a card as starred, or editing its note, in one deck never affects
  the dictionary original or the same word saved in a *different* deck —
  each save is its own copy.
- Code can still answer "is this dictionary word saved anywhere?" by
  comparing `sourceId`, without decks holding shared references to the same
  object.
- `Deck.terms` always holds these deck-owned copies — never a direct
  reference into `dictionaryWords`.

If something about save/star/note behavior seems confusing, this
copy-on-save split is almost always the reason.

## The dictionary pipeline

`assets/dictionary/dictionary.db` is a pre-built SQLite database bundled
with the app — it is **not** built at runtime. It's generated offline by
Python scripts in `tools/`, primarily `convert_jmdict_to_sqlite.py`, which
parses `tools/source/JMdict_e.gz` (a gzipped XML dump of
[JMdict](https://www.edrdg.org/jmdict/j_jmdict.html), the standard
open-source Japanese–English dictionary project) into the schema
`DictionaryService` expects. The other `tools/debug_*.py` and
`check_priority_row.py` scripts are one-off debugging aids used while
building that conversion, not part of the app itself.

**Gotcha worth remembering:** `DictionaryService.dictionaryDatabaseFileName`
is a versioned filename (`dictionary_v7.db` as of this writing), with a
comment instructing you to bump the version whenever you regenerate the
asset. This exists because the app copies the bundled asset into permanent
on-device storage the first time it runs — without bumping the version,
a device that already extracted an older copy would keep using the stale
one indefinitely rather than picking up the new database.
